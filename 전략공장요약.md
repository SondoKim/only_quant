# 전략 공장(Strategy Factory) 구조 분석

## 1. 전체 파이프라인

```
Bloomberg 데이터
    ↓
[Discovery] 조합적 전략 생성 (모멘텀 + 평균회귀 + 고급)
    ↓
[Backtest] VectorBT 배치 백테스트 (멀티프로세싱)
    ↓
[Filter] Sharpe 3Y ≥ 1.0 & Sortino 3Y ≥ 1.0 & 거래수 ≥ 20
    ↓
[Store] src/factory/strategies_YYYY-MM-DD/ (월말 폴더)
    ├── MOM_XXXXXX.json (개별 전략)
    ├── MR_XXXXXX.json
    ├── ADV_XXXXXX.json
    └── index.json (메타데이터 인덱스)
    ↓
[Select] 상관관계 필터링 + Sharpe 6M 가중 포트폴리오
    ↓
[Signal] 자산별 포지션 집계 → 매매 신호
```

## 2. 핵심 모듈 구조

| 모듈 | 역할 |
|------|------|
| `src/strategies/generator.py` | 파라미터 조합으로 전략 **전수생성** (MA, RSI, MACD, Z-score, 스프레드 등) |
| `src/strategies/momentum.py` | 모멘텀 신호 생성 (MA교차, EMA, RSI, MACD, 돌파, 스프레드모멘텀) |
| `src/strategies/mean_reversion.py` | 평균회귀 신호 (Z-score, RSI극단, 볼린저, 스프레드Z, 스프레드퍼센타일) |
| `src/strategies/advanced.py` | 고급 전략 7종 (필터모멘텀, Lead-Lag, 멀티TF, 변동성돌파, 상대강도, ADX, 캐리) |
| `src/backtester/vectorbt_engine.py` | VectorBT 기반 벡터화 백테스트 + 성과지표 계산 |
| `src/factory/strategy_factory.py` | JSON 저장/로드/필터/인덱스 관리 |
| `src/portfolio/selector.py` | 상관관계 그리디 필터링 + Sharpe 가중 집계 → 자산별 포지션 |
| `src/indicators/` | 기술적/크로스자산 지표 (벡터화) |
| `config/assets.yaml` | 자산 정의 + 크로스자산 페어 (커브, 크로스컨트리, 캐리) |
| `config/indicators.yaml` | 파라미터 범위 + 백테스트 임계값 |

## 3. 실행 모드 (main.py)

- **`discover`**: 전수/샘플링 전략 생성 → 백테스트 → 필터 통과 시 저장
- **`signals`**: 최신 전략폴더에서 활성 전략 로드 → 오늘의 매매 신호
- **`update`**: 일일 업데이트
- **`summary`**: 공장 현황 요약

## 4. Walk-Forward 백테스트 (`scripts/run_backtest.py`)

**2단계 시뮬레이션:**
1. **Phase 1**: 각 월말마다 discovery가 없으면 실행 (전략폴더 생성)
2. **Phase 2**: 매일 전일까지의 데이터로 신호 생성 → PnL 계산
   - 드로다운 스케일링, PnL MA 필터 옵션

---

## 5. 장점

1. **체계적 알파 탐색**: 조합적 전수조사로 사람의 편향 없이 전략 발굴
2. **Walk-Forward 구조**: 월별 폴더 분리로 미래 데이터 누수(look-ahead bias) 방지
3. **다양한 알파 소스**: 모멘텀 + 평균회귀 + 고급 = 서로 보완적인 전략군
4. **크로스자산 전략**: 스프레드/비율 기반으로 상대가치 기회 포착
5. **상관관계 필터**: 중복 전략 제거 → 실제 분산 효과 확보
6. **Sharpe 가중 집계**: 최근 성과 좋은 전략에 높은 비중 → 적응적
7. **벡터화 + 멀티프로세싱**: 수만 개 전략 빠르게 테스트 가능
8. **JSON 개별 저장**: 전략 단위 관리/추적 용이

## 6. 단점 및 개선점

### A. 과적합(Overfitting) 리스크 — **가장 큰 구조적 위험**

- **문제**: 수만 개 조합에서 Sharpe 3Y ≥ 1.0을 찾는 것은 "데이터 마이닝"의 전형적 사례. 충분히 많은 조합을 시도하면 랜덤 전략도 필터 통과 가능
- **개선안**:
  - **다중검정 보정** (Multiple Testing Correction): Bonferroni, BH-FDR, 또는 White's Reality Check / Hansen's SPA 적용
  - **Out-of-Sample 검증**: 3년 In-Sample로 발굴 후, 별도 1년 OOS에서 재검증 통과해야 저장
  - **부트스트랩 p-value**: 각 전략의 Sharpe가 통계적으로 유의한지 검정

### B. 전략 퇴화(Decay) 관리 부재

- **문제**: 한번 저장된 전략은 `is_active` 플래그만으로 관리. 시장 레짐 변화 시 더 이상 유효하지 않은 전략이 계속 활성 상태
- **개선안**:
  - **Rolling OOS 성과 추적**: 발굴 이후 실시간 성과를 모니터링하여 임계값 미달 시 자동 비활성화
  - **반감기(Half-life) 메커니즘**: 오래된 전략일수록 가중치 감소
  - **레짐 감지**: 변동성/상관관계 레짐 변화 시 전략 풀 자동 재구성

### C. 포지션 사이징의 단순함

- **문제**: Sharpe 가중 후 [-1, 1] 클램핑만 적용. 변동성 정규화, 리스크 버짓팅 없음
- **개선안**:
  - **역변동성 사이징**: 자산별 변동성에 반비례하는 포지션 크기
  - **리스크 패리티**: 각 자산의 리스크 기여도를 균등하게
  - **켈리 기준(Kelly Criterion)**: 기대수익/분산 기반 최적 비중

### D. 전략 간 상관관계 측정의 한계

- **문제**: `_filter_by_correlation`에서 수익률 상관관계만 사용. 포지션이 같은 방향인데 진입/청산 타이밍만 다른 전략은 상관관계가 낮게 나올 수 있음
- **개선안**:
  - **포지션 상관관계**: 수익률 대신 포지션(시그널) 자체의 상관관계 측정
  - **클러스터링**: DBSCAN 또는 계층적 클러스터링으로 전략 그룹화 후 그룹당 대표 선택

### E. 트랜잭션 비용 모델의 단순함

- **문제**: 고정 bps 슬리피지만 적용. 실제 글로벌 매크로는 자산별 유동성/스프레드가 매우 다름
- **개선안**:
  - **자산별 거래비용**: 선진국 금리선물 vs KRW 현물의 비용 차이 반영
  - **시장 충격 모델**: 포지션 크기에 비례하는 비용
  - **롤오버 비용**: 선물 만기 롤 비용 반영

### F. 저장 구조의 확장성

- **문제**: 2,800개 × 28개월 = ~78,000개 JSON 파일. 디스크 I/O와 인덱싱에 비효율
- **개선안**:
  - **SQLite/DuckDB**: JSON 대신 경량 DB로 전환. 쿼리 속도 대폭 향상
  - **Parquet**: 전략 메타데이터를 월별 parquet 파일로 통합
  - 현재 `index.json`이 사실상 DB 역할을 하고 있으므로, 이를 proper DB로 승격

### G. 발견(Discovery)의 비효율

- **문제**: 매월 전수조사 시 대부분 파라미터는 이전 달과 동일한 결과. 불필요한 중복 계산
- **개선안**:
  - **증분 Discovery**: 이전 달 대비 변경분만 재계산 (새 데이터 1개월분의 영향만 평가)
  - **Bayesian Optimization**: 전수조사 대신 베이지안 최적화로 유망 영역 집중 탐색
  - **Early Stopping**: 백테스트 중간에 Sharpe가 임계값에 도달 불가능하면 조기 중단

### H. 리스크 관리 보강

- **문제**: 드로다운 스케일링이 옵션이고 단순. 테일 리스크 관리 없음
- **개선안**:
  - **VaR/CVaR 제약**: 포트폴리오 레벨 손실 한도
  - **스트레스 테스트**: 과거 위기 시나리오 (2008, 2020 COVID, 2022 금리충격) 에서의 성과 확인
  - **동적 레버리지**: 시장 변동성 레짐에 따른 전체 익스포저 조절
